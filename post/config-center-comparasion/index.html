从框架支持来看，可以依据对 spring framework 的支持分出 3 个 ： diamond, Disconf 和 Apollo。由于它们需要支持 springframework，难免引入一些自有注解，对应用代码有较强侵入性。这类框架在一些大公司得到采用的原因是，它们既要兼容老的 spring 框架，又要允许新业务采用 Spring Boot。

如果团队仅使用 SpringBoot，这类框架应谨慎考虑。

持久和容灾

持久的选择往往决定了容灾级别。 MySQL 之类的 RDMS 都不易扩展，有概率发生单点故障。如果客户端直接和服务端交互，单点服务同样会有可用性隐患，因此许多配置中心均采用本地冗余的方式提高容灾。目前，在存储层真正达到易部署、多节点高可用的，只有 etcd 和 nacos。

Apollo 为达到高可用，不惜引入了 Eureka 服务群，但为保证框架流行度又使用 RDMS 作为持久层，局限明显，携程内部早已弃用。此外，diamon, Disconf, Config Kepper 普遍采用了 MySQL 作为持久层。

nacos 与注册中心强绑定，而且对于 SpringBoot 的集成仍处于孵化阶段，这是选择基于 etcd 开发配置中心的原因。

更新模型和性能

推模型优于拉模型，增量更新优于全量更新，另一个要注意的是配置更新的顺序（Happen Before）。

基于 zookeeper 、 etcd 的配置中心 Disconf, Apollo 和 Nacos 均可保证更新事件推送的顺序性和仅更新增量。

Config Keeper 和 Spring Git 如果使用增量更新，需要依赖于 Spring Bus。基于 RabbitMQ 的 Spring Bus 无法保证消费顺序，发生接连更新的情况，如果配置之间存在依赖关系，容易导致非常严重的业务异常。Config Keeper 和 Spring Git 可以通过暴露 refresh endpoint 做全量更新使顺序得到保证，更新模型也就从增量变成了全量。

配置回滚

较新的配置中心基本都支持配置回滚

配置灰度

配置灰度功能的实现，要么需要采用注册中心模式（续租），要么就是客户端轮询，本质上都是注册实例。Appollo 在这方面支持最好，Nacos 是注册中心 + 配置中心，所以应该有支持。目前自研配置中心通过灰度 endpoint 的方式在动态更新的基础上支持了手工灰度。后续项目组有需求，服务端可以从注册中心获取实例信息并做灰度推送。

配置共享

大部分均支持，Spring Git 无显式支持

实例配置监控

基于 SpringBoot Actuator 模块或者 Environment 抽象很容易做到实例的配置监控，所以较新的都有该功能。未来服务数量增多的情况下，服务端可以基于该功能做配置的推送验证，并集成监控。
