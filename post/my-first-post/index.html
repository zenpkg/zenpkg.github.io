约定

约定俗成，我们目前采用如下 JSON 结构进行数据通信

1
2
3
4
5

{
	  "code":0,
	  "msg": "成功",
	  "data": {}/[]
}

设计初衷已无从讲起，好处在于可以区分逻辑错误和业务错误。

当业务正常时，业务系统返回 http 200-300, code 0 ，msg 成功，并在 data 中放置业务数据。

当发生业务错误时（如参数异常，数据不存在，请求非法），业务系统可以返回 200-300 ，code 1(或系统相关错误码), msg “错误原因”, data 置空。

当发生逻辑错误时，如 http 协议异常，缺乏认证头，http 保文格式错误，服务器过载，系统返回 http 400-5xx，http body 为空（这些由 web 容器缺省处理，我们目前少有定制）。

有的厂采用了其他做法区分业务正常和业务异常，正常时返回 http 200，并在 http body 中直接写 data，业务异常时返回 http 400 并在 http body 中返回含有若干错误原因和纠正方法的字段。

约定的烦恼

许多项目在实现时，为了包装或解包装这一数据结构，使用了 BaseResponse, Result, ResultDTO 等名字的 POJO（也有使用 Map 或者 JsonObject)，无论使用哪一种结构，应用一般都需要这么做：
